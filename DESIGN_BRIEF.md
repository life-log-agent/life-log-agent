## 프로젝트 한 줄 설명

`life_log`는 갤러리에 흩어진 **이미지 기록**(사진·스크린샷·캡처)을 모아, AI가 자동으로 내용을 추출·분류하고 **자연어로 검색·질의(RAG)** 하게 해 주는 **반응형 PWA 라이프로그 서비스**다.

## 주요 사용자

자신의 기록을 여기저기 저장만 하고 다시 찾지 못하는 **개인 사용자**. **폰 = 업로드 기기**(갤러리 다중 선택), **PC = 조회·검색 기기**이며 같은 계정을 공유한다. 비전문가 기준의 단순함이 전제.

## 핵심 화면 목록 + 각 화면의 목적 + 입력값/결과값

| # | 화면 | 목적 | 입력값 | 결과값 |
|---|------|------|--------|--------|
| 1 | 로그인 / 온보딩 | 로그인 + 첫 사용 안내 | 계정 인증 | 인증된 세션, 첫 행동 유도 |
| 2 | 업로드 | 폰 갤러리 다중 선택 업로드 | 이미지 파일(다중 선택, `image/*`), 선택 파일 미리보기/개수, (선택) 메모·태그 힌트 | 업로드 접수 확인, 처리 시작 |
| 3 | 처리 상태 | 처리 진행을 명료히 표시 | (없음 — 조회) | 항목별 `pending → processing → ready / failed` 진행, 실패 시 재시도 |
| 4 | 분류 결과 / 타임라인 | 자동 분류 + 시점 기준 조회 | (선택) 카테고리/시점 필터 | 카테고리·태그·엔티티별 묶음, 시간순 타임라인 |
| 5 | 검색 / 질의응답(RAG) | 자연어로 다시 찾기 | 자연어 질의 + 시간/장소/카테고리 필터 | RAG 답변 + **근거 원본 기록**(썸네일·날짜·카테고리·장소), 적용된 필터 |
| 6 | 항목 상세 | 원본과 추출 내용 확인 | 항목 선택 | 원본 미리보기 + 추출 텍스트 + 메타데이터 |

> `captured_at`은 EXIF 자동 추출 — 수동 입력 불필요(없을 때만 보정).

## 3분 발표 최소 흐름: 입력 → 처리 중 → 결과 → 다시 시도

1. **입력:** 업로드 화면에서 합성 이미지 1~수 장 선택 → 업로드 버튼.
2. **처리 중:** 처리 상태 화면에서 `pending → processing → ready` 단계가 눈에 띄게 진행(스피너 단독 아님).
3. **결과:** 검색 화면에서 자연어 질의(예: "사고 싶었던 화장품 보여줘") → RAG 답변 + 근거 썸네일·카테고리.
4. **다시 시도:** 실패(`failed`) 항목은 **재시도 버튼**으로 재처리 / 검색 결과 0건이면 필터 변경·재질의로 다시 시도.

## 로딩 / 오류 / 빈 상태

- **로딩:** 단계형 진행 표시(`pending → processing → ready`). 스피너만 띄우지 않는다.
- **오류:** `failed` 사유 + **재시도 버튼**(파이프라인은 멱등·재처리 가능). 업로드 실패도 재시도 가능.
- **빈 상태:** 기록 0건 → 첫 업로드 유도. 검색 결과 0건 → 다음 행동(필터 변경/업로드) 안내. 빈 화면을 막다른 길로 두지 않는다.

## 화면 밀도

**발표용으로 단순하게.** 단계형(가이드형) 흐름 — 업로드 → 처리 상태 → 분류 결과 → 검색을 명료한 단계로 안내한다. 한 화면에 정보를 욱여넣지 않고, 칸반 보드 같은 고밀도 UI는 피한다. 모바일 우선 반응형 — 텍스트가 겹치거나 잘리지 않게, 가독성 최우선.

## 버튼 우선순위 (가장 먼저 눌러야 할 버튼)

1. **업로드 버튼** — 첫 화면의 1차 액션. 가장 크고 명확하게.
2. **검색 진입** — 2순위.
3. 재시도 / 필터 — 상황(오류·빈 결과)에서 보조적으로.

> 첫 화면에 바로 보여야 할 것: 업로드 1차 액션 + 최근 기록 타임라인 요약 + 검색 진입점. 처리 중 항목이 있으면 진행 상태를 눈에 띄게, 비어 있으면 첫 업로드 유도.

## 모바일에서 가장 중요한 행동

**업로드.** 폰의 1차 역할이 갤러리에서 골라 올리기다. 업로드 버튼을 **엄지가 닿는 하단에 크고 명확하게 고정**한다. 그다음이 검색.

## 더미 데이터 규칙

- 예시·목업은 **명백한 합성 데이터만** 사용한다(실데이터처럼 보이지 않게).
- 예: `cosmetic_lipstick.png`, `jeju_travel.png`, `food_restaurant.png` 같은 합성 파일명과 가짜 썸네일.
- 카테고리 예시: 화장품 / 여행지 / 맛집 / 기타.
- 인물·장소·가격은 가공 값으로, 실존 인물·실제 업체·실제 가격을 쓰지 않는다.

## ⚠️ 주의 — 실제 개인정보·API 키 금지

- **실제 개인정보(PII)를 넣지 않는다.** 실명·전화번호·주소·계좌·결제 스크린샷 등 진짜 개인정보를 목업·예시·캡처에 사용하지 않는다.
- **API 키·토큰·비밀번호를 넣지 않는다.** 화면·목업·문구에 실제 키/토큰/쿠키/개인키를 넣지 않는다. 표기가 필요하면 `<redacted>`로 대체한다.
- 디자인 산출물(목업 이미지 포함)에도 위 값이 드러나지 않게 한다.
