## 1. 프로젝트 요약

**life_log** — 갤러리에 흩어진 사진·스크린샷·캡처를 한곳에 모으면, AI(Clova HCX-005)가 내용을 자동으로 OCR·분류하고, 나중에 **자연어로 검색·질의(RAG)** 할 수 있는 **반응형 PWA + FastAPI** 라이프로그 서비스.

- **한 줄**: "여기저기 저장만 하고 다시 못 찾는" 개인 기록을, 찾을 수 있는 기록으로.
- **대상 사용자**: 스크린샷·사진을 쌓아두지만 다시 못 찾는 개인. 폰=업로드 기기, PC=조회·검색 기기(같은 Supabase 계정 공유).
- **대표 시나리오(데모 기준)**: 제주 흑돼지 사진 업로드 → 자동 분류·인덱싱 → "지난달 제주 흑돼지 식당 어디였지?" → 답변 + 근거 사진.
- **현재 상태**: 실서버(NCloud) 배포 완료 · 백엔드 테스트 15 passed · 배포 전 보안 점검(IDOR 차단·PII 로그 제거) 반영.
- **스택**: React PWA(Vite+TS) / FastAPI / Supabase(Postgres+pgvector·Storage·Auth) / Clova Studio HCX-005·Embedding.

## 4. 장표별 핵심 메시지 · 넣을 자료

### S1 — 표지
- **메시지**: "쌓이기만 하던 기록을, 찾을 수 있는 기록으로."
- **자료**: 서비스명 `life_log`, 한 줄 정의, 팀/발표자. 부제: *AI가 OCR·분류하고 자연어로 검색하는 PWA*.
- **금지**: 과장 카피("세상을 바꾸는", "혁신적인") 금지. 담백하게.

### S2 — 문제 & 대상 사용자 (Before)
- **메시지**: "그 캡처 어디 갔지?" — 저장은 하는데 다시 못 찾는다.
- **자료**: 대상 사용자 1줄, 문제 1줄, 폰=업로드/PC=검색 사용형태. Before 이미지: 어수선한 갤러리 썸네일 그리드(합성).

### S3 — 솔루션 & 대표 시나리오 (After)
- **메시지**: 자연어 한 줄로 끝. *"사고 싶던 화장품 캡처 모아줘" / "지난달 제주 흑돼지 식당?"*
- **자료**: 대표 시나리오를 업로드→답변까지 4스텝 아이콘 흐름. 비목표(음성·PDF·자동수집 제외) 1줄로 범위 명확화.

### S4 — 데모 화면 워크스루 ★디자인 핵심
- **메시지**: 단계형 흐름으로 누구나 따라 한다.
- **자료(실제 캡처 5장, 모바일 폭)**: ①홈(타임라인) ②업로드(갤러리 다중 선택) ③처리 상태(`pending→processing→ready`) ④검색 결과(답변 + 근거 사진) ⑤항목 상세(원본+추출 텍스트·메타).
- **추가**: 빈/실패 상태 작게 1줄(HomeEmpty·ProcessingFailed·SearchEmpty 존재).
- **주석**: 하단 고정 업로드 버튼·모바일 우선·단계형 흐름을 캡처에 화살표로.

### S5 — 시스템 아키텍처 & Ncloud 구성 ★Ncloud 핵심
- **메시지**: 무거운 데이터는 매니지드, 앱은 NCloud — 단순·견고.
- **자료(연결 다이어그램 1장)**:
  ```
  [React PWA] ──업로드 직행──> [Supabase Storage]
       │  JWT                         │ Signed URL
       ▼                              ▼
  [FastAPI · NCloud 서버] ──> [Clova HCX-005 (비전)]  · OCR/설명/분류/RAG
       │                  └─> [Clova Embedding v2]   · 1024차원
       ▼
  [Supabase Postgres + pgvector]  · 메타 + 벡터
  ```
- **왜 이 구성**: Clova=한국어 OCR·이해 품질 + 비전 멀티모달이라 OCR 엔진 분리 불필요 / Embedding=LLM과 동일 제공자 정렬 / NCloud=앱 상시 구동(systemd+Nginx+HTTPS), 데이터는 Supabase 매니지드.

### S6 — AI 기능: 파이프라인 4단계
- **메시지**: 업로드와 처리를 분리, 모든 입력을 **공통 텍스트 표현**으로 정규화 후 임베딩.
- **자료**: `업로드(202·백그라운드) → HCX-005 OCR·설명 → 분류(카테고리·태그·장소·요약) → 임베딩→pgvector(ready) → 검색(유사도+필터+재랭크)`. 상태머신 `pending→processing→ready/failed`(failed 재시도) 강조.

### S7 — AI 입출력 실제 예시 & 실패 처리 ★AI 핵심
- **메시지**: 입력·출력이 구체적이고, **실패해도 갇히지 않는다.**
- **자료(실제 1쌍)**: 데모 이미지 → `describe_image` 출력(`OCR:…/설명:…`) → `classify` JSON(`{category,tags,place,summary}`) → RAG 출력(`{answered,text}` + evidence 썸네일).
- **실패 처리 4종(반드시 표기)**: ①LLM JSON 파싱 실패 → 안전 폴백(`기타`/`answered:false`) ②타임아웃 60s ③OCR 텍스트 없으면 임베딩 skip ④파이프라인 예외 → `status=failed` + 재시도.
- **환각 방지**: 유사도 0.30 미만 컷 + "등록된 기록만 근거, 출처 없으면 답 안 함".

### S8 — (선택) 라이브 데모 / 데모 영상
- **메시지**: 말보다 동작. 10~20초 클릭 흐름.
- **자료**: 실서버 라이브 또는 GIF(업로드→ready→검색→답변). 실패 시 대비해 S4 캡처가 백업.

### S9 — 테스트 & 오류 해결
- **메시지**: 동작을 검증하고, 실배포 버그를 근본 원인까지 추적·수정했다.
- **자료(검증)**: `pytest 15 passed` · `ruff clean` · 외부 API는 `integrations/` 모킹. 표: test_auth 7(JWT 회귀) / test_ingest 6(202·큐잉·**타인경로 403·traversal 403**·retry) / test_search 2. E2E 실연동 증거 1컷(업로드→`ready`).
- **자료(오류 해결, 발표 스토리)**: ①`7d606dd` captured_at naive/aware 버그 → `processing` 영구정지 근본원인 추적·수정·서버반영 ②`7c09115` 배포 전 보안점검에서 **IDOR 자가 발견 → 본인 경로 강제 차단** + PII 로그 제거. (정직성: mypy는 모듈명 충돌로 미실행 — LOW, 후속.)

### S10 — 보안 원칙 & 마무리
- **메시지**: 라이프로그=민감정보 多 → 처음부터 보안 설계.
- **자료**: 키는 환경변수만(코드/문서/로그 비노출) · service_role 백엔드 전용/anon만 프론트 · JWT(ES256/JWKS) 신뢰경계 · PII 로그 미출력 · 테스트 외부호출 모킹. 후속 1줄: **RLS/Storage 정책 적용 예정**(정직 표기). 마무리: 현재 상태 + 다음 단계.

## 5. AI 기능 · Ncloud 구성 · 화면 디자인 흐름 (Design 참고)

- **AI 기능 흐름(S6·S7)**: 4단계 파이프라인을 **좌→우 수평 플로우**로. 각 단계 위에 모델/출력 라벨. 실패 경로는 점선으로 `→ failed → 재시도`. 입출력 예시는 **코드/JSON 모노스페이스 카드**로(실물 느낌).
- **Ncloud 구성(S5)**: 박스+화살표 다이어그램. NCloud 영역과 Supabase(매니지드) 영역을 **배경색으로 구분**. 화살표 라벨: 업로드 직행 / Signed URL / JWT / 임베딩.
- **화면 디자인 흐름(S4)**: 폰 목업 프레임 안에 실제 캡처 5장을 **데모 순서대로 가로 나열**. 단계 번호(①~⑤)와 짧은 캡션. 모바일 우선·하단 고정 CTA를 화살표 주석으로.

## 6. 테스트 / 오류 해결 장표 내용 (S9 상세)

- **상단**: 큰 숫자 `15 passed` + `ruff clean` 배지. "외부 API 모킹 — 실호출 없음" 한 줄.
- **좌**: 테스트 분해 표(파일/검증/개수). 보안 테스트(403 2종)는 강조색.
- **우**: 오류 해결 타임라인 2건(captured_at 버그, IDOR 차단) — 각 `증상 → 원인 → 수정 → 반영` 4칸.
- **하단**: 정직성 1줄(mypy 미실행 사유 + RLS 후속). 과장 없이 "발견·수정·검증" 사이클을 보여주는 게 목적.

## 7. 디자인 톤

- **실제 앱 데모 화면처럼.** 추상 일러스트·스톡 이미지보다 **실제 캡처/목업** 우선. 화면이 주인공.
- **과한 마케팅 금지.** 거대 카피, 과장 형용사("혁신·최고·세계 최초"), 불필요한 그라데이션·애니메이션 금지.
- **가독성 최우선.** 저대비 텍스트·작은 폰트 금지(WCAG 대비 확보). 한 장 한 메시지, 정보 욱여넣기 금지.
- **단계형(가이드형) 인상.** 흐름이 번호로 읽히게. 모바일 우선 톤.
- **일관된 시각 언어.** 파이프라인/아키텍처 다이어그램은 같은 박스·화살표·색 규칙 재사용.
- **합성 데이터만.** 목업·예시는 명백한 합성 데이터(실명·전화·계좌 금지).

## 8. 민감정보 금지 규칙 (전 장표 공통)

- **실 IP·호스트네임·계정**(`root@…`, `…nip.io`) 노출 금지 → `<SERVER_IP>`로 마스킹.
- **키·시크릿**: 변수명만 OK, 실제 값 절대 금지. 데모 중 `.env`/터미널 env/`/docs` 인증 토큰 노출 금지.
- **JWT 토큰**: 슬라이드/캡처/개발자도구 네트워크 탭에 노출 금지(캡처 전 개발자도구 닫기).
- **PII**: 데모는 **전용 계정 + 합성/비민감 이미지**만. 캡처본의 이메일·실 사진 내용 마스킹.
- **git 화면**: `git log` 등 히스토리(과거 실 IP 잔존 가능) 화면공유 금지.
- **발표 직전 1분 점검표**: ☐ `.env`/env 안 보임 ☐ 개발자도구 닫힘 ☐ 실 IP/호스트 안 보임 ☐ 데모 계정·합성 이미지 ☐ `git log`/토큰 미노출.
