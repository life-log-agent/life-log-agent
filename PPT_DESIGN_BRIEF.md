## 1. 프로젝트 요약

- **프로젝트명**: life_log
- **한 줄 설명**: 갤러리에 흩어진 사진·스크린샷·캡처를 AI가 자동으로 OCR·분류하고, 자연어로 검색·질의(RAG)하는 반응형 PWA.
- **대상 사용자**: 기록을 여기저기 저장만 하고 다시 못 찾는 개인. 폰=업로드 기기, PC=조회·검색 기기(같은 Supabase 계정 공유).
- **해결하려는 문제**: 스크린샷·사진이 갤러리에 쌓이지만 "그거 어디 저장했더라"로 다시 못 찾는다.
- **발표에서 보여줄 MVP 흐름** (대표 시나리오: 제주 흑돼지):
  - **입력**: 폰 갤러리에서 사진 여러 장 선택 → 업로드(즉시 202 응답).
  - **처리 중**: `pending → processing → ready` 상태가 화면에 단계로 표시. HCX-005가 OCR·설명 → 분류 → 임베딩.
  - **결과**: *"지난달 제주 흑돼지 식당 어디였지?"* → AI 답변 + **근거가 된 원본 사진**(썸네일·날짜·카테고리) 함께 표시.
  - **다시 시도 / 기록**: 처리 실패 시 `failed`로 떨어지고 **재시도 버튼** 제공. 모든 기록은 타임라인·상세로 남음.

## 5. 장표별 핵심 메시지 · 넣을 자료

### S1 — 표지
- **메시지**: "쌓이기만 하던 기록을, 찾을 수 있는 기록으로."
- **자료**: 서비스명 `life_log`, 한 줄 정의, 팀/발표자. 부제 *AI가 OCR·분류하고 자연어로 검색하는 PWA*.
- **금지**: 과장 카피("세상을 바꾸는", "혁신적") 금지.

### S2 — 문제와 사용자 (Before)
- **메시지**: "그 캡처 어디 갔지?" — 저장은 하는데 다시 못 찾는다.
- **자료**: 대상 사용자 1줄, 문제 1줄, 폰=업로드/PC=검색 사용형태, 어수선한 갤러리 그리드(합성 이미지).

### S3 — MVP 범위 (실제 동작하는 기능 흐름)
- **메시지**: 이번 발표에서 **실제로 동작**하는 건 이만큼이다(과장 없이).
- **자료**: 대표 시나리오를 **입력→처리중→결과→재시도** 4스텝으로. 비목표(음성 STT·PDF·자동 갤러리 수집 제외) 1줄로 범위 명확화.

### S4 — AI · Ncloud 시스템 구성 ★최대 배점(30)
- **메시지**: **AI·클라우드가 하나의 사용자 흐름으로 맞물린다.** 무거운 데이터는 매니지드, 앱은 NCloud — 단순·견고.
- **자료(연결 다이어그램 1장, 쉽게 라벨링)**:
  ```
  [폰: React PWA] ──①사진 직행 업로드──> [Supabase Storage]
        │ ②경로+메타(JWT 검증)                 │ ④Signed URL
        ▼                                       ▼
  [FastAPI · NCloud 서버] ──③──> [Clova HCX-005 (비전)]  OCR·설명·분류·RAG 답변
        │                  └──> [Clova Embedding v2]    텍스트→1024차원 벡터
        ▼ ⑤저장/검색
  [Supabase Postgres + pgvector]   메타데이터 + 벡터 인덱스
  ```
- **번호 흐름 캡션(쉬운 설명)**: ①폰이 원본을 Storage에 직접 올림 ②백엔드엔 경로만 전달(로그인 토큰 검증) ③AI가 사진을 읽고 분류·임베딩 ④검색 시 잠깐 열리는 안전한 이미지 링크 ⑤벡터로 "비슷한 기록" 검색.
- **왜 이 구성(이유 명시 — Ncloud 설계자 관점)**:
  - **Clova HCX-005**: 한국어 OCR·이해 품질 우수 + **비전 멀티모달이라 OCR 엔진을 따로 안 붙여도 됨** → 파이프라인이 단순.
  - **Clova Embedding v2**: LLM과 **같은 제공자로 정렬**, 한국어 검색 품질 + 1024차원을 pgvector에 그대로 저장.
  - **NCloud 서버(Ubuntu)**: 앱을 systemd로 상시 구동(Nginx 리버스 프록시 + HTTPS). 데이터 플랫폼은 Supabase 매니지드로 운영 부담↓.
- **인프라 한 줄(인프라 팀장 관점)**: 단일 공인 IP 서버 1대 + Supabase. 비밀값은 서버 환경변수로 주입(`.env` 평문 미배포).

### S6 — AI 입출력 실제 예시 & 실패 처리 ★최대 배점(30)
- **메시지**: 입력·출력이 **구체적**이고, **실패해도 갇히지 않는다.**
- **자료(실제 1쌍, 모노스페이스 카드)**:
  - 입력: 데모 이미지 1장 →
  - `describe_image` 출력: `OCR: <추출 텍스트> / 설명: <한국어 설명>` →
  - `classify` 출력: `{category, tags(≤5), place, summary}` (category는 화장품/여행지/맛집/기타) →
  - RAG 출력: `{answered: bool, text}` + evidence(썸네일·날짜·점수).
- **AI 실패 처리 4종(반드시 표기 — AI 엔지니어 관점)**: ①LLM JSON 파싱 실패 → 안전 폴백(`기타`/`answered:false`) ②타임아웃 60s ③OCR 텍스트 없으면 임베딩 skip ④파이프라인 예외 → `status=failed` + **재시도 가능**.
- **환각 방지**: 유사도 0.30 미만 컷 + "등록된 기록만 근거, 출처 없으면 답하지 않음".
- **데이터 매니저 관점**: 예시는 전부 **합성/데모 데이터**임을 라벨로 명시.

### S5 — 디자인 흐름 (입력 → 처리 중 → 결과 → 다시 시도)
- **메시지**: 단계형(가이드형) 흐름 — 누구나 순서대로 따라 한다.
- **자료(실제 캡처, 모바일 폭)**: ①홈(타임라인) ②업로드(갤러리 다중 선택) ③처리 상태(`pending→processing→ready`) ④검색 결과(답변+근거 사진) ⑤항목 상세. 빈/실패 상태(HomeEmpty·ProcessingFailed·SearchEmpty) 작게 1줄.
- **주석**: 하단 고정 업로드 버튼·모바일 우선을 캡처에 화살표로.

### S7 — 구현 자료 (폴더 구조·실행·연결 화면)
- **메시지**: 말이 아니라 **돌아가는 코드**다.
- **자료**: 레이어 구조(Routers→Services→Integrations→Models/DB) 한 줄 + 폴더 트리 요약. 실행 화면(`uvicorn` 기동/`/docs`), 실서버에서 업로드 1건이 `ready`까지 간 `GET /items` 응답 1컷(= 실연동 증거). API 표(`/ingest`·`/items`·`/search`…).

### S8 — 테스트 & 오류 해결
- **메시지**: 동작을 검증하고, 실배포 버그를 근본 원인까지 추적·수정했다.
- **자료(검증)**: `pytest 15 passed` · `ruff clean` · 외부 API는 `integrations/` 모킹. 표: test_auth 7(JWT 회귀) / test_ingest 6(202·큐잉·**타인경로 403·traversal 403**·retry) / test_search 2.
- **자료(오류 해결 — 증상→원인→수정→반영)**: ①`7d606dd` captured_at naive/aware 버그 → `processing` 영구정지 해결, 예외 시 rollback 후 `failed`로 재시도 가능 ②`7c09115` 배포 전 보안점검에서 **IDOR 자가 발견 → 본인 경로 강제** + PII 로그 제거.
- **정직성**: mypy는 모듈명 충돌로 미실행(LOW, 후속).

### S9 — 한계와 개선 방향
- **메시지**: 지금 못한 것과 다음에 할 것(과장 없이).
- **자료**: 한계 — 입력은 이미지 단일 모달리티(음성·PDF 미지원), 자동 갤러리 수집은 브라우저 제약으로 불가, **RLS/Storage 정책 미적용(후속)**. 개선 — RLS 적용, 모달리티 확장(파이프라인 2단계에 처리기 추가), 검색 필터 고도화.

### S10 — 마무리 (만든 것 + 제출 산출물)
- **메시지**: 무엇을 만들었는지 + 제출물.
- **자료**: 현재 상태(실서버 배포 완료·15 passed·보안 점검 반영). 산출물 목록: 코드 저장소, `README.md`, `SERVER_RUNBOOK.md`, 발표 자료. 다음 단계 1줄.

## 6. AI 기능 · Ncloud 구성 · 화면 흐름 (Design 참고 — 한 흐름으로 연결)

> **30점 항목의 핵심은 "셋이 따로"가 아니라 "하나의 사용자 흐름"으로 보이는 것.** S4(구조)에서 셋을 한 장에 묶고, S5(화면)·S6(입출력)이 그 구조의 단계를 각각 보여주는 식으로 연결한다.

- **AI 기능 흐름(S6)**: 4단계 파이프라인을 **좌→우 수평 플로우**. 각 단계 위 모델/출력 라벨. 실패 경로는 점선 `→ failed → 재시도`. 입출력은 모노스페이스 JSON 카드로 실물감.
- **Ncloud 구성(S4)**: 박스+화살표. **NCloud 영역**과 **Supabase(매니지드) 영역**을 배경색으로 구분. 화살표에 번호+쉬운 라벨(업로드 직행/경로 전달/AI 처리/Signed URL/벡터 검색).
- **화면 흐름(S5)**: 폰 목업 프레임에 실제 캡처 5장을 **데모 순서대로** 가로 나열. 단계 번호(①~⑤)+짧은 캡션. S4 다이어그램의 번호와 **같은 번호 체계**를 써서 "구조 ↔ 화면"이 눈으로 연결되게.

## 7. 테스트 / 오류 해결 장표 내용 (S8 상세)

- **상단**: 큰 숫자 `15 passed` + `ruff clean` 배지. "외부 API 모킹 — 실호출 없음" 한 줄.
- **좌**: 테스트 분해 표(파일/검증/개수). 보안 테스트(403 2종) 강조색.
- **우**: 오류 해결 2건을 `증상 → 원인 → 수정 → 반영` 4칸 카드로(captured_at 버그, IDOR 차단).
- **하단**: 정직성 1줄(mypy 미실행 사유 + RLS 후속). 과장 없이 "발견·수정·검증" 사이클이 목적.

## 8. 디자인 톤

- **실제 앱 데모 화면처럼.** 추상 일러스트·스톡보다 **실제 캡처/폰 목업** 우선. 화면이 주인공, 정보 밀도 유지(마케팅 랜딩페이지 X).
- **과한 마케팅 금지.** 거대 카피·과장 형용사("혁신·최고·세계 최초")·불필요한 그라데이션/애니메이션 금지.
- **쉬운 이해 우선.** 전문용어(RAG·pgvector·임베딩)는 **한 줄 비유/캡션**을 함께. 심사위원이 처음 봐도 30초 내 파악.
- **가독성 최우선.** 저대비·작은 폰트 금지(WCAG 대비). 한 장 = 제목 1 + 핵심 메시지 1 + 자료 2~4개.
- **일관된 시각 언어.** 파이프라인/아키텍처/화면 흐름에 **같은 번호·박스·화살표·색 규칙** 재사용(구조↔화면 연결).
- **작은 화면 가독성.** 표·코드 카드도 축소 화면에서 읽히게(폰트·여백 확보).

## 9. 민감정보 금지 규칙 (전 장표 공통)

- **실 IP·호스트네임·계정**(`root@…`, `…nip.io`) 금지 → `<SERVER_IP>`로 마스킹.
- **키·시크릿**: 변수명만 OK, 실제 값 금지. 데모 중 `.env`/터미널 env/`/docs` 인증 토큰 노출 금지.
- **JWT 토큰**: 슬라이드·캡처·개발자도구 네트워크 탭 노출 금지(캡처 전 개발자도구 닫기).
- **개인정보·금융/의료/학생 정보**: 데모는 **전용 계정 + 합성/비민감 이미지**만. 실제 PII 절대 금지.
- **샘플/실데이터 구분(데이터 매니저)**: 모든 예시 데이터에 "샘플/데모" 라벨.
- **git 화면**: `git log` 등 히스토리(과거 실 IP 잔존 가능) 화면공유 금지.
- **발표 직전 1분 점검표**: ☐ `.env`/env 안 보임 ☐ 개발자도구 닫힘 ☐ 실 IP/호스트 안 보임 ☐ 데모 계정·합성 이미지 ☐ `git log`/토큰 미노출.

## 11. Claude Design 결과 검수 기준

- 첫 3장 안에 "무엇을 만들었는지"가 보이는가
- **AI·Ncloud·화면 디자인이 하나의 사용자 흐름으로 연결**되어 설명되는가 (★30점)
- 실제 입력과 결과 화면이 보이는가
- 오류 수정 또는 테스트 결과가 빠지지 않았는가
- 장표가 보기만 좋고 평가표 확인 자료가 빠져 있지 않은가
- 작은 화면에서도 표와 코드 카드가 읽히는가
- 전문용어가 쉬운 캡션/비유로 풀려 있는가
- 제출 전 README·실행 방법·개선 방향이 정리되어 있는가
- 실 키·서버 정보·개인정보가 없고 모든 데이터에 "샘플/데모" 표시가 있는가
