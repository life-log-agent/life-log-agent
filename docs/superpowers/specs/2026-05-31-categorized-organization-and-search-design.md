# life_log — 카테고리 자동 정리 & 자연어 검색 설계

- 작성일: 2026-05-31
- 상태: 브레인스토밍 완료, **사용자 스펙 리뷰 대기** (다음 단계: writing-plans)
- 상위 설계: `docs/superpowers/specs/2026-05-31-life-log-platform-design.md`
- 범위: 제품 기능 정의를 구체화/정제. 플랫폼 아키텍처(PWA·FastAPI·Supabase·Clova)는 상위 설계를 따르되, **본 문서가 입력 모달리티·분류·정리 기능의 최신 기준**이다.

## 1. 한 줄 요약

갤러리에 마구잡이로 캡처해 둔 **이미지/스크린샷**을 올리면, AI가 **카테고리별로 자동 정리**해 주고(장소는 지도 링크, 쇼핑은 제품 카드), 나중에 **자연어로 검색**할 수 있다.

대표 시나리오:
- "저번에 스크린샷 찍어둔 부산 맛집 정보 가져와줘" → 관련 캡처 + 장소 카드
- "사고 싶었던 화장품들 보여줘" → 쇼핑/위시리스트 카드 목록

## 2. 상위 설계 대비 변경점 (이번 브레인스토밍 결정)

| 항목 | 상위 설계(platform-design) | 본 문서(확정) |
|------|----------------------------|----------------|
| 입력 모달리티 | 이미지 + 음성메모(STT) + PDF | **이미지/스크린샷만** (STT·PDF 제거) |
| Web Share Target | 안드로이드 한정 점진 향상 | **제거** (iOS 미지원 → 모든 기기 공통 위해 제외) |
| 분류 | 카테고리·태그 자동 분류(열린) | **하이브리드** — 핵심 카테고리는 고정+맞춤 화면, 나머지는 "기타" |
| 카테고리별 정리 | 명시 없음 | **카테고리별 맞춤 정리/액션** 추가 (본 문서 §4) |

> 음성·PDF·Web Share Target은 폐기가 아니라 **현재 범위에서 제외**. 추후 필요 시 `integrations`/파이프라인에 처리기 추가로 흡수 가능.

## 3. 입력 & 처리 흐름

```
이미지/스크린샷 다중 선택 업로드 (폰 1차, PC 가능)
  → Storage 저장 + Item(status=pending) 생성 (즉시 응답, 백그라운드 처리)
  → HCX-005 비전: 설명 + OCR 텍스트
  → 분류: category 판정 (장소/맛집 · 쇼핑/위시리스트 · 기타)
  → 카테고리별 구조화 필드 추출 (§4)
  → 텍스트 임베딩 → pgvector 저장 → status=ready
  → 카테고리 뷰 / 자연어 검색에서 조회
```

- 업로드 엔드포인트는 무거운 작업을 동기 수행하지 않는다(상위 설계 불변 규칙 유지).
- `Item.status`(`pending→processing→ready/failed`)가 진행의 단일 진실원, 처리는 멱등.

## 4. 분류 & 카테고리별 정리 (핵심)

**분류 모델: 하이브리드.** AI가 항목을 아래 핵심 카테고리 중 하나, 또는 "기타"로 판정한다.

**데이터 추출 모델: 방식 1 — 카테고리별 정해진 필드(스키마).**
AI는 `category` + 그 카테고리에 정의된 구조화 필드를 반환한다. "기타"는 구조화 없이 설명 텍스트만 둔다. (자유형 key-value 방식은 카드 일관성·정렬 약화로 채택하지 않음.)

### 4.1 장소/맛집
- 추출 필드: `place_name`(필수), `address`(있으면)
- 정리 화면: 장소 카드 목록. 각 카드에 **"네이버지도에서 열기" 딥링크**.
- 네이버지도 **개인 즐겨찾기 자동 추가는 불가** (공개 쓰기 API 없음 — 확인 완료). 딥링크로 사용자가 직접 저장하는 반자동 방식.

### 4.2 쇼핑/위시리스트 (화장품 포함)
- 추출 필드: `product_name`, `brand`, `price`(추출 가능 시)
- 정리 화면: 제품 카드 목록(제품명·브랜드·가격). **사용자 메모** 추가 가능.

### 4.3 기타
- 구조화 필드 없음. 설명/OCR 텍스트만 보유.
- 카테고리 맞춤 화면 없이 **자연어 검색(RAG)으로 커버**.

## 5. 검색 (자연어 / RAG)

- 전체 항목 대상 자연어 질의 → 질의 임베딩 → pgvector 유사도(+ 카테고리/시점 메타 필터) → HCX-005 재랭크·답변 합성.
- 결과: 관련 캡처 썸네일 + 해당 카테고리 카드(장소면 딥링크, 쇼핑이면 제품 정보).

## 6. 데이터 모델 (개념)

`Item` (상위 설계 기반, 본 문서 반영):
- 공통: `id`, `user_id`, `filename`, `storage_path`, `status`, `extracted_text`, `embedding`, `category`, `captured_at`, `created_at`
- 카테고리별 구조화 필드는 `category_data`(JSON)에 스키마대로 저장:
  - 장소/맛집: `{ place_name, address? }`
  - 쇼핑/위시리스트: `{ product_name, brand?, price? }` + 사용자 `memo`
  - 기타: 없음

## 7. 스코프 경계

**포함:** 이미지 업로드, 비전 추출, 하이브리드 분류, 장소(딥링크)·쇼핑(카드+메모) 정리, 자연어 RAG 검색, Supabase Auth/저장.

**제외(현재 범위 밖):** 음성메모(STT), PDF, Web Share Target, 네이버지도 개인 즐겨찾기 자동 추가, 앱 내 지도 뷰, 구매여부/위시리스트 상태 토글.

## 8. 미해결 / 다음 단계에서 정할 것

- 핵심 카테고리 추가 기준(새 카테고리 스키마 추가 절차).
- 가격 파싱 신뢰도/통화 표기 처리.
- 분류 모호 항목("기타" vs 핵심) 판정 임계.
- 구현 사이클 분할(슬라이스 1과의 접합) — writing-plans에서 구체화.
