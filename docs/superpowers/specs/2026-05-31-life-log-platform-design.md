## 1. 제품 개요

`life_log`는 **AI 라이프로그 에이전트**다. 사용자가 갤러리에 모아둔 캡처·사진·스크린샷 등 **이미지**(사고 싶은 화장품, 가보고 싶은 여행지, 맛집 등)를 업로드하면, AI가 내용을 추출·자동 분류하고, 나중에 **자연어로 검색·질의**할 수 있게 한다.

대표 사용 시나리오:
- "사고 싶었던 화장품 캡처들만 모아줘"
- "지난달 제주도 여행 때 먹었던 흑돼지 식당 뭐였지?"

## 2. 핵심 결정과 근거

### 2.1 플랫폼: 반응형 PWA (웹 단독)

**결정:** 네이티브 앱·하이브리드 앱을 만들지 않고, **반응형 PWA(Progressive Web App) 하나**로 데스크톱·iOS·안드로이드를 모두 지원한다.

**근거 (브레인스토밍에서 도출):**
- 사용자 1순위 제약은 **"웹 단독 운영"** 이며, 개발 장비는 **LG그램(Windows), 맥 없음**이다.
- PWA는 윈도우에서 개발·배포가 끝나고, 갤럭시·아이폰·PC 모두 브라우저로 접속해 "홈 화면에 추가"로 앱처럼 쓴다. **맥·Xcode·앱스토어·Apple Developer 계정($99)·스토어 심사 전부 불필요.**
- "맥 없이 아이폰 지원"이라는 요구가 PWA 선택으로 자동 충족된다.

### 2.2 명시적으로 범위에서 제외한 것

- **갤러리 자동(백그라운드) 수집:** 브라우저 보안 정책상 불가능. 어떤 웹 기술로도 사용자가 가만히 있는 동안 갤러리를 빨아들일 수 없다. → 자동 수집은 포기하고 "저마찰 수동 업로드"로 대체.
- **네이티브 동기화 앱:** "웹 단독" 제약과 충돌하므로 제외. (추후 필요 시 동일 백엔드에 얹어 확장 가능하나 현재 범위 밖.)
- **텔레그램 봇 등 메신저 인제스천:** 사용자 결정으로 제외.
- **MYBOX / Google Photos 등 외부 클라우드 자동 연동:** MYBOX는 제3자 사진 읽기 공개 API 없음, Google Photos는 전체 라이브러리 직접 읽기 제한(Picker 반자동만) → 자동 수집 수단으로 부적합하여 제외.
- **음성메모(STT)·PDF 입력(2026-06-06):** 현재 범위에서 제외하고 입력을 이미지 단일 모달리티로 좁힌다. 파이프라인이 모달리티 확장형이므로 필요해지면 단계 2에 처리기를 추가해 흡수한다(아키텍처 변경 없음).

### 2.3 역할 분담

- **폰** = 업로드 기기 (갤러리에서 사진 골라 올림)
- **PC** = 조회·검색·정리 기기
- 둘은 **같은 클라우드 계정**을 공유한다. "폰→PC 파일 이동" 단계는 존재하지 않는다 (폰 브라우저에서 직접 업로드).

## 3. 아키텍처

### 3.1 컴포넌트

| 컴포넌트 | 역할 | 기술 |
|----------|------|------|
| PWA 프론트엔드 | 업로드 UI, 타임라인/카테고리 뷰, 자연어 검색·RAG 대화, 인증, 설치형/오프라인 셸 | React + Vite + TypeScript, 서비스워커 |
| 백엔드 API | 업로드 접수, 인제스천 파이프라인, 벡터 검색, JWT 검증 | FastAPI (순수 JSON API) |
| 데이터 플랫폼 | Postgres+pgvector(메타+임베딩), Storage(원본), Auth(로그인) | Supabase |
| 멀티모달 LLM (비전) | 이미지/스크린샷/캡처 OCR·설명, 자동 분류, RAG 답변 합성 | Clova Studio HCX-005 |
| 임베딩 | 텍스트 임베딩 | Clova Studio Embedding |

### 3.2 수집(인제스천) 경로 — 웹 단독 제약 하

1. **1차 (전 플랫폼):** 폰/PC 브라우저에서 **갤러리 다중 선택 업로드**. `<input type="file" accept="image/*" multiple>` 로 iOS·안드로이드 모두 갤러리(스크린샷 앨범 포함) 다중 선택 지원. 클라이언트가 Supabase Storage에 직접 업로드 후 백엔드 `/ingest` 호출(저장 경로 + JWT).
2. **점진 향상 (안드로이드 한정):** PWA **Web Share Target** 등록 → 갤러리에서 "공유 → life_log". iOS Safari는 미지원이므로 progressive enhancement로만 취급(없어도 1차 경로로 동작).

### 3.3 처리 파이프라인 (5단계)

```
1. 업로드 (routers/ingest)
   - Storage에 원본 저장 → Item 레코드 생성(status=pending)
   - 즉시 202 응답, 처리는 백그라운드 위임

2. 처리 (services/pipeline)
   - 이미지/스크린샷/캡처 → HCX-005 비전: 설명 + OCR 텍스트
   - (음성메모 STT·PDF는 현재 범위 밖 — 필요 시 단계 2에 처리기 추가로 흡수)
   ↓ 공통 텍스트 표현으로 정규화

3. 이해/분류 (services/classify)
   - HCX-005가 카테고리·태그·엔티티(장소/인물/제품/음식)·시점(captured_at) 추출

4. 인덱싱 (services/index)
   - 정규화 텍스트 청크 → 임베딩 → pgvector 저장 → status=ready

5. 검색 (routers/search + services/search)
   - 질의 임베딩 → 벡터 유사도 + 메타 필터(시간/장소/카테고리)
   - 상위 결과를 HCX-005로 재랭크·답변 합성 (RAG)
```

**불변 규칙:**
- 업로드 엔드포인트는 무거운 작업(LLM/임베딩)을 동기 수행하지 않는다. 항상 백그라운드 위임.
- 모든 모달리티는 단계 2에서 **공통 텍스트 표현**으로 수렴. 새 모달리티 추가 = 단계 2에 처리기 하나 추가.
- `Item.status`(`pending → processing → ready / failed`)가 진행의 단일 진실원. 처리는 멱등.

### 3.4 레이어 경계

`Routers`(HTTP+JWT 검증) → `Services`(도메인 로직) → `Integrations`(HCX-005/Embedding/Storage 래퍼) → `Models`(SQLModel) → `DB`(Supabase 세션). 외부 SDK/REST는 `integrations/`를 통해서만 호출.

## 4. 데이터 흐름 (조회·검색, PC 주 사용)

자연어 질의 → 임베딩 → pgvector 유사도 + 메타데이터 필터 → HCX-005 재랭크/답변 합성. UI는 카테고리별 뷰(화장품/여행지/맛집…)와 타임라인 제공.

## 5. 에러 처리 & 엣지 케이스

- **업로드 실패:** 클라이언트 재시도, 부분 업로드 정리.
- **처리 실패:** Item `status=failed` + 재처리 가능(멱등 보장: item_id 기준 청크/임베딩 삭제 후 재삽입).
- **중복 업로드:** 파일 해시로 dedup.
- **대용량 이미지:** 업로드 전/후 리사이즈.
- **iOS PWA 한계 명시:** Web Share Target·백그라운드 동기화 미지원 — 1차 업로드 경로로 동작, 사용자 안내 문구 제공.

## 6. 보안 & 프라이버시

- **신뢰 경계 = Supabase JWT.** 보호 엔드포인트는 JWT 검증 후에만 `user_id` 사용. 클라이언트가 보낸 `user_id` 불신. Supabase RLS 병행.
- service role 키는 백엔드 전용, 프론트엔드에는 anon 키만.
- 이미지(사진·스크린샷·캡처)는 민감정보 다수 → 외부 전송(Clova/Supabase) 고지·동의, 로그에 원본 내용/추출 텍스트/식별자 미출력.
- 시점은 UTC(timezone-aware) 저장, EXIF `captured_at` 우선 추출.

## 7. 테스트 전략

- 백엔드: pytest — 파이프라인·검색 단위, 외부 AI는 `integrations` 경계에서 모킹(실제 API 미호출).
- 프론트: Vitest + React Testing Library.
- e2e: 업로드 → 분류 → 검색 happy path.

## 8. 개발/배포 환경

- 개발 장비: Windows(LG그램) 단독으로 전 과정 가능. 맥 불필요.
- 프론트 배포: 정적 호스팅 + HTTPS(PWA 요건). 백엔드: Python 호스트(Supabase는 관리형).

## 9. 향후 확장 (현재 범위 밖, 참고)

- 자동 수집이 핵심 가치로 재부상하면, 동일 백엔드에 얇은 네이티브 동기화 앱(React Native+Expo, EAS 클라우드 빌드로 맥 없이 iOS 빌드)을 추가하는 경로가 열려 있다.
